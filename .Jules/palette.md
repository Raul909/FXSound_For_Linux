## 2024-03-06 - Improve accessibility for icon-only buttons and labeled selects
**Learning:** In React implementations where generic `<div>`s are used as visual labels next to interactive elements, screen readers will completely fail to associate them unless explicitly linked using `id` and `aria-labelledby`. Also, icon-only buttons that convey binary state (like power) must combine `aria-label` with `aria-pressed` to correctly indicate both function and current status to assistive technologies.
**Action:** When adding or reviewing custom interactive components (like dropdowns and status toggles), verify that visual text is explicitly linked to `<select>` or `<input>` tags using proper ARIA attributes, and ensure all purely visual icon buttons have corresponding textual alternatives and explicit state announcements.

## 2025-03-07 - Add full keyboard and screen reader accessibility to custom `<div>` sliders
**Learning:** When building custom slider components using `<div>` elements (e.g. `EQBand.jsx` and `EffectSlider.jsx`), relying solely on mouse events (`onMouseDown`, `mousemove`) completely excludes keyboard users and assistive technologies. A generic `<div>` has no semantic meaning or focus behavior by default.
**Action:** Ensure all custom visual interactive components implemented with `<div>` tags are made natively accessible. For sliders, this requires explicitly providing `role="slider"`, `tabIndex` for focusability, `aria-valuenow`/`aria-valuemin`/`aria-valuemax` for screen reader value reporting, and an `onKeyDown` handler to capture standard keyboard navigation (Arrow keys, Home, End). Additionally, implement a distinct `:focus-visible` styling state to visually indicate keyboard focus.
